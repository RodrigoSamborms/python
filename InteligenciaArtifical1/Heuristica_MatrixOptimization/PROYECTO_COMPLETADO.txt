OPCIÓN 4: Uso Programático

  from plan_optimizer import PlanOptimizer
  import numpy as np

  matrices = {
      'A': np.random.randn(500, 500),
      'B': np.random.randn(500, 500),
      'C': np.random.randn(500, 500),
      'D': np.random.randn(500, 500),
  }

  optimizer = PlanOptimizer(
      expression="(A @ B) + (C @ D)",
      matrices=matrices,
      population_size=20,
      generations=30
  )

  best_plan, stats = optimizer.optimize()
  print(f"Mejor plan: {best_plan}")
  print(f"Mejora: {stats['improvement']:.2f}%")

🎯 CONCEPTOS CLAVE
════════════════════════════════════════════════════════════════════════════

DAG (Directed Acyclic Graph)
  └─ Grafo de dependencias entre operaciones
  └─ Identifica qué se puede paralelizar

ExecutionPlan (Cromosoma)
  └─ use_parallel: ¿paralelizar o no?
  └─ num_workers: cuántos workers
  └─ size_threshold: cuándo paralelizar
  └─ scheduling_strategy: qué estrategia

Fitness
  └─ Medida de calidad: tiempo de ejecución (ms)
  └─ Menor = mejor plan

Operadores Genéticos
  ├─ Selección: torneo (elige mejores)
  ├─ Cruce: combina genes de dos padres
  ├─ Mutación: modifica genes aleatoriamente
  └─ Elitismo: preserva mejores soluciones


💡 CASOS DE USO
════════════════════════════════════════════════════════════════════════════

✓ Matrices pequeñas (< 200x200)
  → AG recomienda: SERIAL
  → Razón: Overhead de paralelismo no compensa

✓ Matrices medianas (200-500x500)
  → AG recomienda: PARALELO (2-3 workers)
  → Razón: Beneficio empieza a superar overhead

✓ Matrices grandes (> 500x500)
  → AG recomienda: PARALELO (4+ workers)
  → Razón: Beneficio >> overhead

✓ Expresiones complejas
  → AG optimiza el plan globalmente
  → No solo decisiones binarias


📈 VELOCIDAD DE CONVERGENCIA
════════════════════════════════════════════════════════════════════════════

Típicamente:
  Generación 1-5:   Mejora rápida (población aleatoria → mejores planes)
  Generación 5-15:  Mejora moderada (explotación local)
  Generación 15-30: Mejora lenta (convergencia a óptimo local)

Con población=15 y generaciones=20:
  Tiempo total: ~30-60 segundos (matrices 300x300, 2 ejecuciones/plan)
  Mejora típica: 5-20% vs baselines


🔬 ASPECTOS TÉCNICOS
════════════════════════════════════════════════════════════════════════════

Representación:

Operadores:
  ├─ Selección: Torneo tamaño 3

🚀 PRÓXIMAS MEJORAS (Hipotéticas - Fase 3)
════════════════════════════════════════════════════════════════════════════

   → Convergencia más rápida

   → Adaptación a nuevos hardware


❓ FAQ
════════════════════════════════════════════════════════════════════════════

P: ¿Cuánto tiempo tarda el AG?
R: Depende de población, generaciones y tamaño de matrices.
   Típico: 30-120 segundos para matrices 300-500x500

P: ¿Necesito más generaciones para mejor resultado?
R: Generalmente retornos decrecientes después de generación 20.
   Probar: población=20, generaciones=15-25

P: ¿Puedo usar matrices aún más grandes?
R: Sí, pero el AG tardará más. Aumentar `num_runs` si quieres precisión.

P: ¿El AG siempre encuentra el óptimo global?
R: No, es un algoritmo aproximado. Busca óptimos locales.
   Ejecutar múltiples veces da resultados diferentes.

P: ¿Cómo cambio los parámetros del AG?
R: En main_ga.py el usuario ingresa valores,
   O edita plan_optimizer.py para valores por defecto.

P: ¿Por qué el AG a veces elige SERIAL aunque hay paralelismo?
R: Porque el overhead de sincronización > beneficio computacional.
   Esto es una conclusión CORRECTA del AG.
